# Blueprint Document Template

Write to `docs/TWIN_BLUEPRINT.md` (or the user's chosen path). Every
section required, in order. Source claims carry `file:line`; platform and
store claims carry a docs URL + date fetched during this run.

---

```markdown
# App Blueprint: <product name> mobile twin

> Generated by twin-my-site on <date>. Driver: <why an app, one line>.
> Platforms: <iOS / Android / both>. Scope: <e.g. reader app; admin stays web>.

## 1. Scope and drivers

The driver, target platforms, scope verdict with rationale, non-goals
(what stays web-only and why), team constraints that shaped decisions.

## 2. Feature matrix

The full census table: feature, audience, evidence, verdict
(include/adapt/exclude/mobile-new), mobile shape, API needs.
Counts: <N include / N adapt / N exclude / N mobile-new>.

## 3. API: audit, gaps, contract

- **Existing surface:** what the audit found (`file:line`), including
  "server-rendered only, no API" if true.
- **Gap list:** every MISSING endpoint with its domain-code reuse pointer —
  flagged as server-side roadmap work.
- **Contract:** per-resource endpoint tables (auth, versioning, pagination,
  media variants, sync fields, push pipeline).

## 4. Design translation

- Mobile token set (with web sources).
- Component map (web → app, per-platform notes).
- Open design decisions with recommendations (dark mode, icon, motion,
  font licensing status).

## 5. Stack decision

- **Decision:** <the ONE stack>
- **Rationale:** team skills / web-stack affinity (verified claim) /
  census feature needs / maintenance reality
- **Rejected:** <alternative — one line each>

## 6. Mobile-native layer

Offline model (what's readable offline, sync strategy, storage budget),
push pipeline end to end, deep/universal links mapping web URLs to app
screens, analytics parity.

## 7. Store compliance

The CURRENT rules that apply, each with citation URL + date checked:
minimum functionality, IAP applicability for this product's monetization,
account deletion, privacy labels / data safety. Scope impacts highlighted.

## 8. Phased roadmap

Phase 1 is the **walking skeleton**: real auth + one content list + one
detail screen, end-to-end through the real API on a device. Each phase
sized for one deep-plan run:

### Phase <N>: <name> (magnitude: S/M/L)
- **Scope:** concrete screens/endpoints/tokens
- **Server-side prerequisites:** which API gaps must land first
- **Exit criteria:** verifiable (a flow works on device, a store checklist passes)
- **Next:** run deep-plan on this phase

## 9. Risk register

| Risk | Likelihood | Impact | Mitigation |
<font licensing, store rejection vectors, API gap slippage, image
bandwidth, team ramp-up on the chosen stack…>

## 10. Corrections & unverified

Beliefs corrected during analysis (mandatory), and UNVERIFIED items with
what would verify them.
```

---

## Quality bar

- The web team can start building the API gaps from section 3 alone.
- A designer can build the theme from section 4 alone.
- Phase 1 can be handed to deep-plan today.
- Nothing in section 7 is from memory — every store rule was fetched this run.
